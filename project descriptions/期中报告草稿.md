## 第一部分：题目介绍

### 选题背景

互联网上存在海量的公开信息，但这些信息分散在不同网站，缺乏统一的管理和检索手段。手动收集效率低下，且难以对数据进行结构化存储和分析。网络数据爬取管理系统通过自动化抓取、集中存储、统一检索，解决上述问题。

### 项目目标

设计并实现一个完整的网络数据爬取管理系统，具备以下能力：

- 根据用户指定的目标网址，自动抓取网页文本内容和图片信息
- 将爬取数据分类存储至关系型数据库（MySQL）和文档数据库（MongoDB）
- 提供关键字、时间、发布者等多维度检索功能
- 提供友好的 Web 管理界面，支持任务管理、数据查看和统计分析

### 题目来源

课程项目可选题目 · 第一类 · 数据库应用系统 · 三、网络数据爬取管理系统

## 第二部分：需求分析

### 2.1 用户角色

本系统面向单一管理员角色，管理员可执行所有操作，包括发起爬取、管理数据、执行检索。

### 2.2 功能性需求

**爬取管理**

- 用户输入目标 URL，提交爬取任务
- 系统在后台自动执行爬取，用户无需等待
- 用户可查看任务列表、任务状态（待执行 / 进行中 / 已完成 / 失败）

**数据管理**

- 查看已爬取的网站列表和网页列表
- 查看网页对应的正文内容和图片列表
- 删除指定网页及其关联内容（级联删除）
- 导出检索结果为 CSV 文件

**内容检索**

- 按关键字搜索正文内容
- 按时间范围筛选（发布时间 / 爬取时间）
- 按发布者筛选
- 结果分页展示，支持排序

**图片管理**

- 展示所有爬取到的图片缩略图
- 按图片描述关键字筛选
- 查看图片所属网页来源

**统计分析**

- 仪表盘展示：任务总数、网站总数、内容总数、图片总数
- 按网站统计爬取数量
- 按日期统计爬取趋势

### 2.3 非功能性需求

| 类别     | 要求                                             |
| -------- | ------------------------------------------------ |
| 可用性   | 界面操作直观，无需培训即可使用                   |
| 稳定性   | 爬取失败时系统不崩溃，记录错误信息并更新任务状态 |
| 可维护性 | 前后端分离，代码结构清晰，有注释和接口文档       |
| 可部署性 | 使用 Docker Compose 一键启动，无需手动配置环境   |
| 合规性   | 爬取设置请求间隔（≥2秒），遵守基本的爬虫礼仪     |

### 2.4 系统边界（不做的事）

- 不支持多用户 / 权限管理（单管理员）
- 不做登录认证（课程范围内不要求）
- 不做以图搜图等图像内容分析
- 不做实时爬取（任务异步执行，非 WebSocket 推送）

## 第三部分：数据库设计

### 3.1 设计原则

- MySQL 满足第三范式（3NF）：消除传递依赖，每个非主属性只依赖于主键
- MongoDB 部分采用文档模型：面向查询设计，内容字段灵活，不预设固定结构
- 两库通过 `webpage_id` 关联，MySQL 为主，MongoDB 为辅

### 3.2 MySQL 表结构

#### Website（网站）

| 字段         | 类型         | 约束               | 说明         |
| ------------ | ------------ | ------------------ | ------------ |
| id           | INT          | PK, AUTO_INCREMENT | 网站 ID      |
| domain       | VARCHAR(255) | UNIQUE, NOT NULL   | 域名         |
| organization | VARCHAR(255) |                    | 所属机构     |
| contact      | VARCHAR(255) |                    | 联系信息     |
| created_at   | DATETIME     | DEFAULT NOW()      | 首次发现时间 |

#### Webpage（网页）

| 字段       | 类型          | 约束               | 说明     |
| ---------- | ------------- | ------------------ | -------- |
| id         | INT           | PK, AUTO_INCREMENT | 网页 ID  |
| url        | VARCHAR(2048) | UNIQUE, NOT NULL   | 网页地址 |
| website_id | INT           | FK → Website.id    | 所属网站 |
| crawl_time | DATETIME      | NOT NULL           | 抓取时间 |
| status     | ENUM          | NOT NULL           | 抓取状态 |
| title      | VARCHAR(512)  |                    | 网页标题 |

#### DataSource（数据源信息）

| 字段         | 类型          | 约束               | 说明         |
| ------------ | ------------- | ------------------ | ------------ |
| id           | INT           | PK, AUTO_INCREMENT | 记录 ID      |
| webpage_id   | INT           | FK → Webpage.id    | 所属网页     |
| publisher    | VARCHAR(255)  |                    | 数据发布者   |
| publish_time | DATETIME      |                    | 发布时间     |
| source_url   | VARCHAR(2048) |                    | 原始数据链接 |

#### CrawlTask（爬取任务）

| 字段        | 类型          | 约束               | 说明                      |
| ----------- | ------------- | ------------------ | ------------------------- |
| id          | INT           | PK, AUTO_INCREMENT | 任务 ID                   |
| target_url  | VARCHAR(2048) | NOT NULL           | 目标起始 URL              |
| status      | ENUM          | NOT NULL           | 待执行/进行中/已完成/失败 |
| created_at  | DATETIME      | DEFAULT NOW()      | 提交时间                  |
| finished_at | DATETIME      |                    | 完成时间                  |
| page_count  | INT           | DEFAULT 0          | 爬取页数                  |
| error_msg   | TEXT          |                    | 错误信息                  |

### 3.3 范式验证（以 Webpage 为例）

**1NF**：所有字段均为原子值，无重复组，满足。

**2NF**：主键为单字段 `id`，不存在复合主键的部分依赖，满足。

**3NF**：检查传递依赖——`website_id → domain`（网站域名）是否应放在 Webpage 表中？否，域名属于 Website 表，Webpage 只保留外键 `website_id`，无传递依赖，满足 3NF。

### 3.4 MongoDB Collection 设计

#### contents（正文内容）

```json
{
  "_id": "ObjectId",
  "webpage_id": 42,
  "webpage_url": "https://example.com/article/1",
  "text_content": "正文全文...",
  "keywords": ["数据库", "爬虫", "Python"],
  "word_count": 1200,
  "crawl_time": "2026-04-01T12:00:00Z"
}
```

#### images（图片）

```json
{
  "_id": "ObjectId",
  "webpage_id": 42,
  "webpage_url": "https://example.com/article/1",
  "image_url": "https://example.com/img/photo.jpg",
  "description": "系统架构示意图",
  "crawl_time": "2026-04-01T12:00:00Z"
}
```

**选用 MongoDB 的理由**：网页正文长度、关键字数量、图片数量因页面而异，字段结构不固定，关系型数据库需要预定义所有字段且不利于全文存储，MongoDB 文档模型更适合此类内容数据。

### 3.5 完整性约束

- `Webpage.website_id` 外键约束：删除 Website 时级联删除其所有 Webpage
- `DataSource.webpage_id` 外键约束：删除 Webpage 时级联删除 DataSource
- `Webpage.url` 唯一性约束：防止同一 URL 被重复爬取
- `Website.domain` 唯一性约束：防止同一域名重复注册
- `CrawlTask.status` ENUM 约束：限制状态值范围

### 4.3 爬取流程模型

```
用户提交 URL
      │
      ▼
FastAPI 创建 CrawlTask（status=pending）
      │
      ▼
BackgroundTasks 触发 Scrapy
      │
      ├── 抓取网页 HTML
      │       │
      │       ├── 解析文本内容 ──→ 写入 MongoDB (contents)
      │       ├── 解析图片 URL ──→ 写入 MongoDB (images)
      │       └── 提取元信息   ──→ 写入 MySQL (Webpage, DataSource)
      │
      ▼
更新 CrawlTask（status=done / failed）
      │
      ▼
前端轮询接口获取最新状态
```

## 第五部分：功能设计

### 5.1 爬取任务管理

**提交任务**

- 输入：目标起始 URL（必填）、爬取深度（可选，默认2层）
- 处理：创建 CrawlTask 记录，后台异步执行 Scrapy Spider
- 输出：返回任务 ID 和初始状态

**查询任务状态**

- 前端每5秒轮询 `GET /api/tasks`
- 展示字段：任务 ID、目标 URL、状态、提交时间、完成时间、爬取页数

**异常处理**

- 网络超时：Scrapy 设置 `DOWNLOAD_TIMEOUT=30`，超时记录错误并继续下一页
- 任务失败：将错误信息写入 `CrawlTask.error_msg`，前端展示失败原因

### 5.2 内容检索

**检索参数**

| 参数       | 类型   | 说明                  |
| ---------- | ------ | --------------------- |
| keyword    | string | 在正文中进行文本匹配  |
| publisher  | string | 精确匹配发布者        |
| start_date | date   | 发布时间下限          |
| end_date   | date   | 发布时间上限          |
| page       | int    | 分页页码（默认第1页） |
| page_size  | int    | 每页条数（默认20条）  |

**检索逻辑**

- 关键字检索：在 MongoDB `contents` 集合的 `text_content` 字段使用正则匹配
- 时间筛选：在 MySQL `DataSource` 表按 `publish_time` 范围查询，获取 `webpage_id` 列表，再到 MongoDB 按 `webpage_id` 过滤
- 发布者筛选：在 MySQL `DataSource` 表按 `publisher` 精确查询

### 5.3 数据管理

**删除逻辑（级联）**

```
删除 Webpage
  → 删除 MySQL: DataSource（外键级联）
  → 删除 MongoDB: contents（按 webpage_id）
  → 删除 MongoDB: images（按 webpage_id）
```

**CSV 导出**

- 检索结果可一键导出为 CSV
- 字段包含：标题、URL、发布者、发布时间、关键字、来源网站

### 5.4 统计分析

- 总览卡片：任务数 / 网站数 / 内容数 / 图片数（实时查询）
- 折线图：近30天每日新增爬取页数
- 柱状图：各网站爬取页数排行（Top 10）

## 第六部分：模块划分

### 6.1 模块结构

```
网络数据爬取管理系统
│
├── 前端模块（Vue 3 + Element Plus）
│   ├── 仪表盘页面
│   ├── 爬取任务页面
│   ├── 内容检索页面
│   ├── 图片管理页面
│   ├── 数据管理页面
│   └── 公共组件（导航栏、分页、加载状态）
│
├── 后端 API 模块（FastAPI）
│   ├── 任务路由（tasks）
│   ├── 检索路由（contents, images）
│   ├── 数据管理路由（websites, webpages）
│   ├── 统计路由（stats）
│   └── 数据访问层（MySQL + MongoDB 连接封装）
│
├── 爬虫模块（Scrapy）
│   ├── GeneralSpider（通用网页爬虫）

└── 基础设施模块（Docker Compose）
    ├── MySQL 服务
    ├── MongoDB 服务
    ├── FastAPI 服务
    └── Vue 前端服务
```
